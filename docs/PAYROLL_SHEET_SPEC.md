### 0. Global conventions (lock these as invariants)

- **Proration divisor is a fixed 30**, never calendar days. Every sheet uses `*Q/30` where `Q` = Days Worked; even 31-day May uses 30. Name it `DAYS_IN_PAYROLL_MONTH = 30`.
- **Full IEEE float precision is stored; round only at display/disbursement.** The current DB columns are `numeric(14,2)` (`supabase/migrations/0002_seed_april_2026.sql`) which CANNOT round-trip values like medical `9090.90909…`, WHT `204.5454545…`, `O18=239856.0606060606`, `O40=936124.2515151515`. **Change money columns to `numeric` (no scale) or `numeric(18,6)`.** The code `r2()` rounding in `engine.ts` must be removed from all intermediate steps.
- **Recompute-on-read invariant:** never trust a stored headline; provide a pure `recompute(record)` that regenerates gross/taxable/wht/net from inputs `(D, days, OT, Inc, Bonus, otherDeductions, advance)`.
- **Entity is a tag dimension in the ONE shared database** — JU (JU Estimation), PDC (Pavilion Design Consultants), B4U. Do NOT split databases.
- **Workbook TOTAL rows are NOT reliable** — they hardcode/omit member rows (see §1.1 bugs). The app must re-sum all members and FLAG disagreements, not copy the spreadsheet ranges.

### 1. AS-IS logic per sheet (exact formulas + control totals)

#### 1.1 Department salary sheets — `Admin & HR`, `Sales and Marketing`, `Technical Department`

Identical per-employee engine. Columns: D=Salary, E=Basic, F=Medical, G=Overtime, H=Incentive, I=Bonus, J=Gross, K=Taxable, L=WHT, M=Other Deductions, N=Advance, O=Net, P=Arrears, Q=Days Worked, W=Date of Joining, X=Probation Completion.

| Field | Exact formula | Note |
|---|---|---|
| **Medical (F)** | `F = D * 10/110` (= D/11), tax-exempt | `Admin & HR!F3 = D3*10/110 => 9090.909…` |
| **Basic (E)** | `E = D - F = D * 100/110` | E+F always reconstitutes D |
| **Gross (J)** | `J = SUM(E:I) * Q/30` — **the SUM range INCLUDES medical (F)**, so `J = (D + OT + Inc + Bonus) * Q/30`. At full month, no OT, **J = D** (NOT D−medical). | `Sales!J3 = SUM(E3:I3)*Q3/30 => 130000`; `Tech!J10 => 161333.33 = 220000*22/30` |
| **Taxable (K)** | `K = D - F + SUM(G:I)` — **full-month, NOT prorated by days** | `Admin!K3 = D3-F3+SUM(G3:I3) => 90909.09` |
| **WHT (L)** | `L = FBR_annual(K*12)/12 * Q/30` | annualize full-month taxable, slab tax, de-annualize, then prorate by days |
| **Net (O)** | `O = J - L - M - N` (gross − WHT − other deductions − advance). **Arrears P is tracked separately, NOT in O.** | `Admin!O11 => 10000` after advance N11=25000, J11=35000 |
| **Probation end (X)** | `X = EDATE(W, n) - 1`, n per-employee (1 or 2 months) | `Admin!X4 = EDATE(W4,1)-1`; `Tech!X22 = EDATE(W22,2)-1` |

**FBR 2025-26 annual salaried slabs** (`FBR_annual(a)`):
- `a ≤ 600,000` → 0
- `≤ 1,200,000` → `(a−600000) × 0.01`
- `≤ 2,200,000` → `6000 + (a−1,200,000) × 0.11`
- `≤ 3,200,000` → `116000 + (a−2,200,000) × 0.23`
- `≤ 4,100,000` → `346000 + (a−3,200,000) × 0.30`
- `> 4,100,000` → `616000 + (a−4,100,000) × 0.35`

### 3. TO-BE architecture

A **derivation-first, recompute-on-read** payroll engine over ONE Supabase database, with **entity (JU/PDC/B4U) as a first-class tag dimension**, an **allocation engine**, and four register surfaces (salary, tax/FBR, incentives, disbursement).

**3.1 Core salary engine (corrected decomposition)**
- Store **Salary D** as the canonical input on `salary_structures` and `payroll_records`. Derive `medical = D*10/110`, `basic = D − medical`. **Delete the fabricated `travel`** from type, decompose, schema.
- `gross = (basic + medical + OT + Inc + Bonus) * days/30 = (D + OT + Inc + Bonus) * days/30`. Proration uses the record's actual `days`, never 30.
- `taxable = D − medical + OT + Inc + Bonus` (full month, not day-prorated).
- `WHT = FBR_annual(taxable*12)/12 * days/30` — keep `FBR_SLABS` exactly as in `engine.ts:84-91` (already correct; lock with a unit test on Aadil/Awais/Yahya).
- `net = gross − WHT − other_deductions − advance`. `arrears` tracked, NOT in net.
- Store money at full precision; round only at display/disbursement. Recompute-test re-derives all three dept grand totals (J/L/O) from raw cells and **flags** the workbook subtotal bugs (Hassan 14,000; mismatched ranges) rather than mimicking them.

**3.2 Multi-entity allocation (the headline upgrade)**
- `entities` master (JU/PDC/B4U); `entity` tag on employee (team default) and snapshotted on payroll/commission/expense rows.
- Team→entity routing: Social Media sub-team → PDC; Design team → PDC; everyone else → JU; B4U as carve-out.
- **Allocation-rule engine** with methods: `single_entity` (IT→JU 100%, WHT→JU), `fixed_pct` (utilities/misc 75/25), `explicit_amount` (Admin 55,000 PDC + 18,000 B4U), `team_routing` (derive from employee/team), plus a **`pre_split` layer** (DHA water halved before allocation). Every rule is **editable config flagged as "policy assumption to confirm"**, never buried in code.
- Invariants: per-line allocations sum to the line amount; per-entity totals reconcile to grand total within epsilon; **completeness check** flags any cost line in no subtotal (catches Transworld 7,436). `entityTotals(month) = {JU, PDC, B4U}` for payroll + opex, folding payroll in WITHOUT double counting (replaces the `expenses.ts` exclude-payroll policy). Surface the PDC 454,952-vs-577,447 variance as two buckets (allocated overhead vs direct PDC project payments).

**3.3 Overtime engine (65% basis)**
- `rate = (contractGross × 0.65) / 176` (176 = 22×8, both configurable). Display the 65%-basic and the bare hourly rate (no multiplier folded in), matching col E.
- `amount = (weekdayHours + weekendHours) × rate × multiplier`; multiplier `normal=1.5 | govt=2 | eid=2.5` per line. `subTotal = amount + bonus + pendingOther` (= L=SUM(I:K)). Full precision.
- Estimation-team-only; date-based **no-OT-during-probation** gate (`probationEnd = EDATE(joinedOn, n) − 1`). Bind OT gross to the **current-month** payroll record and flag the stale prior-month source; reproducing 14,103.25 (honest) vs 15,092.10 (sheet) as a documented variance.

## Appendix A — Headline findings
1. CORRECTED: Gross J = SUM(E:I)*Q/30 and the SUM range INCLUDES medical (col F), so at a full month J equals the FULL salary D, not D−medical. The hypothesis paraphrase '(basic+OT+Inc+Bonus)' would understate gross ~9%; the literal SUM(E:I) is right.
2. Net O = J − L − M − N (gross − WHT − OtherDeductions − Advance). The current app's net = gross − WHT ignores deductions and advances entirely (Bilal: sheet net 10,000 after a 25,000 advance vs app 35,000). Critical overpayment bug at engine.ts:193.
3. The app fabricates a travel = 10%-of-gross allowance that the sheet does NOT have (sheet decomposes only into Basic = D−medical and Medical = D*10/110). Drop travel from engine.ts:130-136, types.ts:47-51, schema.

## Appendix E — Per-domain requirements & divergences (raw audit detail)

### core-salary-engine
The AS-IS hypothesis is largely correct but contains one material error about the gross base, which I corrected by recomputing against the dump. The three department sheets ('Admin & HR', 'Sales and Marketing', 'Technical Department') run an IDENTICAL per-employee engine. Per row: D=Salary (full monthly contract gross); F MedicalAllowance = D*10/110 (tax-exempt); E Basic = D-F (=D*100/110). J GROSS SALARY = SUM(E:I)*Q/30 — and crucially the SUM(E:I) RANGE SPANS COLUMN F (Medical), so J = (Basic+Medical+OT+Inc+Bonus)*Q/30 = (D+OT+Inc+Bonus)*Q/30. At a full month with no OT this equals the full Salary D, NOT D-Medical. The hypothesis's paraphrase '(basic+OT+Inc+Bonus)' omitted Medical and would have understated gross by ~9%; the literal SUM(E:I) is right. K Taxable salary = D-F+SUM(G:I) (full month, NOT prorated). L WHT = FBR_annual(K*12)/12*Q/30 (annualize the full-month taxable, take FBR slab tax, de-annualize, THEN prorate by days). O NET = J-L-M-N (gross minus WHT minus Other Deductions M minus Advance N). Sheet keeps full float precision (no rounding). I recomputed four independent blocks plus all three sheet grand totals from raw cells and reconciled every one to the penny (Admin J18=298,333.33/O18=239,856.06; Sales J40=949,881/O40=936,124.25; Tech J28=1,452,233.33/O28=1,415,426.67), confirming the corrected formula set. The current app (engine.ts) diverges on five points: (1) it fabricates a travel=10%-of-gross allowance that the sheet does NOT have (sheet has only Basic+Medical); (2) generateMonthForRoster hardcodes days=30 so it never prorates gross by actual days worked Q; (3) its taxable base = gross-medical omits the +OT/+Incentive/+Bonus that the sheet's K includes; (4) net = gross-wht ignores Other Deductions (M) and Advance (N) entirely (e.g. office-boy Bilal: sheet net 10,000 after a 25,000 advance vs app 35,000); (5) it uses r2() rounding while the sheet carries full float. Schema also lacks a Salary(D) anchor column, OT/Inc/Bonus-in-gross, Other-Deductions/Advance/Arrears columns, per-employee probation-months, and the JU/PDC/B4U entity tag.

**Requirements:**

- **[P0] Salary-anchored two-part decomposition (remove travel)** — Store Salary D as the canonical input. Derive medical = D*10/110 and basic = D - medical. Eliminate the fabricated travel allowance from SalaryStructure, decomposeSalary, types, and schema. A recomputation of basic+medical must equal D exactly.
- **[P0] Day-prorated gross including medical** — Compute gross = (basic + medical + overtime + incentive + bonus) * days / 30, i.e. (D + OT + Inc + Bonus) * days / 30. Proration must use the record's actual days, never a hardcoded 30. Replace generateMonthForRoster's gross=contractGross with the prorated formula.
- **[P0] Full-month taxable base that includes OT/incentive/bonus** — Compute taxable = D - medical + overtime + incentive + bonus, NOT prorated by days. WHT must annualize this taxable (×12), apply the FBR slab, ÷12, then ×days/30. Fix calcWHT to take the full taxable (not gross-medical) and to add the variable components.
- **[P0] Net = gross - wht - other_deductions - advance** — Net must subtract Other Deductions (M) and Advance (N) in addition to WHT. Wire the deductions/advance values into the net computation in both the generator and any recompute path. Add arrears as a tracked, non-net field.
- **[P1] Exact FBR slab engine (already correct — lock it)** — Keep the FBR 2025-26 salaried slabs exactly: [600k:0%], [1.2M:1%], [2.2M:6000+11%], [3.2M:116000+23%], [4.1M:346000+30%], [inf:616000+35%].
